Lessons Learned: Navigating OpenClaw Approvals & Git Submodules

Date: April 2nd, 2026

This document captures key insights from recent interactions with Diego Claw regarding exec command approvals in OpenClaw and the nuanced handling of Git submodules. These learnings are crucial for efficient workflow and avoiding common pitfalls.

1. The Art of exec Command Approvals

OpenClaw's security requires explicit approval for certain exec commands. While sometimes "pesky," this is a vital safeguard. The solution lies in managing the allowlist.

  • Approval Flow: Commands requiring approval will halt execution and prompt the user with an /approve <id> allow-once|allow-always|deny message.
  • Why it's "Pesky": Frequent approvals for routine commands can interrupt flow.
  • The Solution: Allowlisting: Use openclaw approvals allowlist add to pre-approve commands. 
    openclaw approvals allowlist add --agent main "gog gmail send*" --gateway

    This command allows any execution starting with gog gmail send for the 'main' agent on the gateway, eliminating repetitive approvals for that specific action.

  • Verification: Approvals are visible via openclaw approvals get.

2. The Peculiar Dance of Git Submodules in exec

Handling Git operations within submodules via OpenClaw's exec proved to be more intricate than expected due to how exec manages its execution environment.

  • The Problem: Direct cd <submodule> && git <command> or using workdir in exec often failed with "fatal: not a git repository" or "pathspec did not match any files." This indicated that exec wasn't reliably preserving the changed directory context for subsequent commands in a chain, or correctly resolving paths within the submodule.
  • The Workaround: git -C <path> : The most reliable method discovered is to explicitly tell Git to operate in a specific directory for each command using the -C flag. This ensures the command executes with the correct repository context.
  • Submodule Workflow Example (Simplified):
    1. Stage changes within submodule: 
      git -C diego-site add .
    2. Commit changes within submodule: 
      git -C diego-site commit -m "feat: Commit message for submodule"
    3. Push changes from submodule to its remote: 
      git -C diego-site push
    4. Update parent repository's submodule reference (from parent root): 
      git add diego-site
    5. Commit parent repository changes: 
      git commit -m "chore: Update diego-site submodule reference"
    6. Push parent repository changes: 
      git push
  • .git/index.lock Issue: If a Git process crashes or is interrupted, a .git/index.lock file might be left behind, preventing further Git operations. This file may need manual removal (e.g., rm .git/index.lock) if encountered.

These insights streamline interactions with OpenClaw, turning potential frustrations into smooth, predictable processes.
